# Mini Agent 阶段总结

## 总体进展

截至当前，项目已完成 `Phase 0`、`Phase 1`、`Phase 2` 的目标实现，已经具备一个可本地运行的极简 Python Agent Runtime 雏形。系统围绕“`agent loop + tool use + skills 即插即用`”展开，核心链路已经打通，能够通过 CLI 完成 skill 激活、脚本执行、数据库查询等任务。

当前实现遵循了任务文档中强调的几个关键原则：

- 内置工具保持为 `read / write / bash / activate_skill` 四个
- Agent 保持无状态，请求之间不做历史持久化
- Skill 采用文件系统扫描与按需激活机制
- 核心引擎只负责 LLM 调度、工具执行、skill 加载
- 配置、日志、provider 切换、示例数据库等基础能力已补齐

---

## Phase 0 完成情况

Phase 0 的目标是完成项目骨架与基础文件整理，目前已完成：

- 建立了目标目录结构，包括 `core/`、`tools_builtin/`、`skills/`、`adapters/`、`data/`、`logs/`、`uploads/`、`workspace/` 等目录
- 创建并补齐了基础文件，包括 `.env.example`、`.gitignore`、`config.yaml`、`requirements.txt`、`README.md`、`main.py`
- 明确了统一 bootstrap 顺序：加载 `.env`、读取配置、初始化日志、扫描 skills、注册工具、创建 LLM、创建 Agent
- 为 CLI 与后续 WebUI 预留了统一运行时装配入口，避免两套初始化逻辑分叉

这一步的成果是：项目已经具备清晰稳定的基础结构，后续功能开发可以围绕统一入口持续推进。

---

## Phase 1 完成情况

Phase 1 的目标是先把 CLI 核心 MVP 跑通，目前已完成：

### 1. 配置与日志

- 支持从 `.env` 读取 provider API Key
- 支持从 `config.yaml` 读取当前 provider、model 覆盖、skills 目录、最大迭代轮数、WebUI 配置
- 已实现日志初始化，日志同时输出到控制台和 `logs/agent-YYYY-MM-DD.log`
- 已按要求记录请求、LLM 调用、工具执行、异常信息

### 2. 核心引擎

- `core/llm.py` 已实现统一的 LLM 抽象，兼容 OpenAI-compatible 调用方式
- `core/tools.py` 已实现工具注册、OpenAI function calling schema 导出、工具执行兜底
- `core/skills.py` 已实现 skill 扫描、frontmatter 校验、目录文本生成、skill 正文按需加载
- `core/agent.py` 已实现无状态 agent loop，支持模型调用、tool calls 回填、循环执行与超轮数中止

### 3. 内置工具

- `read` 已支持文本读取、二进制识别、超长截断
- `write` 已限制只能写入 `workspace/`、`uploads/`、`logs/`
- `bash` 已支持命令执行、超时控制、统一输出格式、日志记录
- `activate_skill` 已支持按名称加载 skill 正文，并对不存在的 skill 给出可用列表

### 4. CLI 与示例 Skill

- 已实现单次 CLI 调用和交互式 REPL
- 已完成 `hello-world` skill
- `hello-world` 可通过 `activate_skill` 激活，再由 `bash` 执行脚本完成问候

这一步的成果是：项目已经具备最小可运行闭环，能够从用户输入出发，完成“模型判断 -> 激活 skill -> 调工具 -> 返回结果”的完整流程。

---

## Phase 2 完成情况

Phase 2 的目标是补齐多 provider 支持，并验证“数据库能力下沉到 skill 层”的设计，目前已完成：

### 1. 多 LLM Provider

- 已补齐 `kimi`、`qwen`、`deepseek` 三个 provider 的静态配置
- 支持通过配置或运行参数切换 provider
- 支持 provider 对应 model 覆盖
- 已实现 provider 是否配置完成的检查能力
- 对 DeepSeek 不支持图片输入的情况，已实现降级处理与日志告警

### 2. SQLite 示例数据库

- 已实现 `data/seed_sample_db.py`
- 可自动初始化 `data/sample.db`
- 已创建 `employees` 表并预填 10 条稳定示例数据
- 启动时若数据库不存在，可自动补建，且默认不会无提示覆盖已有数据

### 3. sqlite-sample Skill

- 已完成 `skills/sqlite-sample/SKILL.md`
- 已补充 `references/schema.md` 说明表结构与查询方式
- 已实现 `scripts/query.py`
- 查询脚本只允许 `SELECT`
- 若未显式加 `LIMIT`，脚本会自动补 `LIMIT 100`
- 查询结果以 JSON 输出，便于 Agent 进一步整理与回答

这一步的成果是：项目已经验证了“业务能力不放进核心，而是通过 skill + 脚本下沉”的设计路线，后续接入医院指标、采购、发票解析等业务场景时，可以继续沿用同样模式扩展。

---

## 当前已经具备的能力

目前项目已经可以支持以下使用方式：

- `python main.py cli "消息内容"`：单次执行 Agent 请求
- `python main.py cli --interactive`：进入无状态交互式 REPL
- `python main.py setup`：初始化或检查示例数据库
- 根据用户任务自动触发 skill 激活
- 通过 `bash` 执行 skill 内脚本
- 查询示例员工数据库并返回结构化结果
- 在 Kimi / Qwen / DeepSeek 之间切换模型提供方
